# Java HTTP Server From Scratch

An HTTP/1.1 server built from scratch in **pure Java (JDK only — no Maven/Gradle, no Spring,
no Netty, no external libraries)** to understand what web frameworks hide: sockets, the accept
loop, blocking vs non-blocking I/O, the thread-per-connection model and its C10K ceiling, the
reactor pattern, and virtual threads (Project Loom).

The server is implemented **three ways** - a bounded thread pool, virtual threads, and a
single-threaded NIO event loop — and benchmarked head to head.

## What it does

- Parses HTTP request lines and headers by hand; builds valid responses (correct `Content-Length`, CRLF framing).
- Method + path **routing** with proper `404` / `405`.
- **Keep-alive** connections, **request-body** reading via `Content-Length`, and an idle **read timeout** (a Slowloris/DoS defense).
- **Static file serving** with MIME types and streamed binary (raw-byte) bodies.
- **Path-traversal defense** (canonicalize the resolved path and verify containment — not blocklisting `..`).
- Three concurrency backends: fixed thread pool, virtual threads, and a hand-written NIO reactor
  (non-blocking `Selector`, partial-read accumulation, partial-write draining, and scheduled
  timers so delayed responses never block the event loop).

## Benchmarks

Measured on macOS (Apple Silicon), JDK 21, localhost. Load generated by an async Python client
(`tests/load.py`). Debug logging disabled for measurement.

### The C10K wall (thread-per-connection ceiling)

A naive thread-per-connection server (`new Thread` per connection) crashes under load:

```
~2,020 connections → java.lang.OutOfMemoryError: unable to create native thread
```

This is the macOS per-process thread cap (`kern.num_taskthreads` = 2048) minus the JVM's own
threads. No amount of RAM raises it — it's an architectural limit of mapping one OS thread per
connection. Every design below is a way around this wall.

### Fast path — 8,000 concurrent instant requests

| Architecture        | Wall time | Failed | Threads used            |
|---------------------|-----------|--------|-------------------------|
| Thread pool (200)   | 1.41s     | 0      | 200                     |
| Virtual threads     | 1.28s     | 0      | ~8,000 virtual / ~10 OS |
| NIO event loop      | 1.68s     | 0      | **1**                   |

For fast requests all three are competent — the event loop keeps pace on a single thread.

### Slow-client concurrency - 2,000 simultaneous slow requests

| Architecture        | Wall time | Why                                                        |
|---------------------|-----------|------------------------------------------------------------|
| Thread pool (200)   | **30.2s** | only 200 slow requests run at once — the rest queue (head-of-line blocking) |
| Virtual threads     | 3.4s      | all 2,000 run concurrently                                 |
| NIO event loop      | 3.3s      | all 2,000 concurrent on **one thread**, via scheduled timers |

**~9× faster** for virtual threads and the event loop. This is where the models diverge: the
bounded pool trades crash-safety for head-of-line blocking under slow load, while virtual
threads and the reactor sustain thousands of concurrent slow connections.

## The three architectures

| Model                  | Code style           | Concurrency            | Cost / limit                              |
|------------------------|----------------------|------------------------|-------------------------------------------|
| Thread pool            | simple, blocking     | = pool size            | head-of-line blocking under slow load     |
| Virtual threads (Loom) | simple, blocking     | ~millions              | virtual threads park on I/O (heap, not OS threads) |
| NIO event loop         | complex, non-blocking| thousands on one thread| must never block the loop; hand-written state machines |

**Key insight:** virtual threads give event-loop-scale concurrency with blocking-style code —
the JVM runs a `kqueue`/`epoll` event loop underneath and parks blocked virtual threads on the
heap. The hand-written NIO server implements that same machinery explicitly.

## Running it

Requires JDK 21+ (for virtual threads). No build tools.

```bash
# thread-pool / virtual-thread server (routing, keep-alive, static files, etc.)
javac src/HttpServer.java && java -cp src HttpServer

# NIO event-loop server
javac src/NioServer.java && java -cp src NioServer

# test
curl -v localhost:8080/
curl -X POST --data 'hello' localhost:8080/echo
curl localhost:8080/index.html          # served from www/

# load test
python3 tests/load.py
```

## What I learned

- Why `accept()` blocks, and why one slow client freezes a single-threaded server.
- The C10K problem, reproduced empirically down to the exact OS constant.
- HTTP framing: why `Content-Length` must match the body byte count exactly, and why you can't `readLine()` a body.
- Keep-alive, and why idle read timeouts are a security requirement (Slowloris).
- Path traversal (CWE-22): validate the *resolved destination*, don't sanitize the *input spelling*.
- The reactor pattern from the inside - partial reads/writes, and scheduling instead of sleeping.
- Virtual threads vs event loops: the same scalability, one with simple code and one without.
